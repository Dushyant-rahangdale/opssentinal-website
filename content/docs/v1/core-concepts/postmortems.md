# Postmortems

Postmortems document what happened, why it happened, and how to prevent recurrence.

## When to Write One

- Any P1/P2 incident
- Recurring incidents
- Incidents with significant customer impact

## Create a Postmortem

1. Open a resolved incident
2. Click **Create Postmortem**
3. Fill out the template

## Recommended Template

### Summary

What happened and the business impact.

### Timeline

| Time  | Event                 |
| ----- | --------------------- |
| 14:00 | Alert triggered       |
| 14:05 | Engineer acknowledged |
| 14:15 | Root cause identified |
| 14:30 | Fix deployed          |
| 14:35 | Incident resolved     |

### Root Cause

Technical explanation of the failure.

### Impact

- Affected users
- Duration
- Revenue impact (if applicable)

### Resolution

Steps taken to recover service.

### Action Items

Track follow-ups and owners.

## Sharing

- Internal sharing via Slack or email
- Optional public summary on Status Page

## Best Practices

- Complete within 48 hours.
- Focus on systems, not people.
- Track action items to closure.
